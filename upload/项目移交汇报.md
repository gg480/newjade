## 一、需求概述

### 1.1 项目定位

用户（玉器实体店店主）需要一个面向珠宝玉器店的**单店单用户轻量进销存系统**，核心价值是提供经营决策数据——批次回本状态、品类利润率、压货预警，而非传统的财务报表。系统部署在极空间 NAS 上，通过 Docker 容器化运行，局域网内手机/电脑均可访问。

### 1.2 核心业务需求

**双轨库存模型**是整个系统的核心设计：

- **高货**（唯一性商品）：翡翠手镯、精品挂件等，每件单独定价、单独记录成本
- **通货**（标准化商品）：水晶手串、银饰等，一批进货 N 件，总成本需按算法分摊到每件

**经营闭环**：看数据（利润/压货/回本）→ 决定拿货 → 入库（批次/逐件/定价）→ 销售 → 看数据

## 三、部署架构

### 3.1 Docker 容器化

采用**多阶段构建**，镜像仅 80MB：

```
Stage 1 (node:20-alpine): 构建前端 Vue SPA
    ↓ 产出 dist/ 静态文件
Stage 2 (python:3.11-slim): 复制后端代码 + 前端构建产物
    ↓ 最终镜像
uvicorn main:app --host 0.0.0.0 --port 8000
```

### 3.2 数据持久化设计

容器通过**两个目录挂载**实现数据和配置持久化，更新镜像不会丢失数据：

| 本地路径 | 容器路径 | 内容 |
|---------|---------|------|
| `./data/` | `/app/data` | `jade.db`（数据库）、`images/`（图片）、`barcodes/`（条码） |
| `./config/` | `/app/config` | `.env`（配置文件，首次启动自动生成） |

### 3.3 配置加载机制（双层设计）

```
docker-compose environment 变量  ← 最高优先级
        ↓
config/.env 文件                 ← 持久化配置（首次启动从模板自动生成）
        ↓
系统默认值                       ← 兜底
```

首次启动时，程序自动从 `/app/templates/env.template` 复制到 `/app/config/.env`，后续更新不会覆盖已有配置。

### 3.4 CI/CD 流程

```
git push main (GitHub)
    ↓ 触发
GitHub Actions (.github/workflows/docker-build.yml)
    ↓ 自动
Docker Buildx 构建 + 推送到 Docker Hub (lrunningmjgoat/jadeinventory:latest)
    ↓ 用户操作
极空间 NAS 拉取新镜像 → 停旧容器 → 删旧容器 → 用相同挂载参数重建容器
    ↓
数据/配置完整保留，零配置升级
```

### 3.5 极空间 NAS 部署要点

- 端口映射：`8080:8000`（主机 8080 → 容器 8000）
- 必须同时挂载 `/app/data` 和 `/app/config`
- 首次启动后需修改 `config/.env` 中的 `JWT_SECRET` 为随机字符串
- 访问地址：`http://NAS内网IP:8080`

### 后续开发者需要了解的

1. 前端源码在 `frontend/` 目录，修改后需 `npm run build` 并将产物复制到 `backend/static/` 目录，然后重新构建 Docker 镜像
2. 后端 API 通过 FastAPI 自动生成 Swagger 文档，启动后访问 `/docs` 查看
3. 所有业务参数通过 `sys_config` 表管理，不硬编码在代码中
4. 器型的规格参数由 `spec_fields` JSON 字段驱动，新增器型时只需修改数据库，无需改代码
5. 成本分摊算法有三种（均摊/按克重/按售价比例），分摊后自动触发定价引擎计算建议零售价
6. 套装销售的价格分摊支持两种模式（按售价比例 / 链子按原价剩余归主件）
